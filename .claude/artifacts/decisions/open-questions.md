# Open Questions

## Open

- [x] [Affects: problem-description-slotmap.md] What is the exact structure of `<primary>` / `<secondary>` elements in openMSX machine XMLs for MSX2/2+/turboR? (Answer: Expansion is structural — devices are direct children of `<primary>` when non-expanded; `<secondary slot="0..3">` children appear when expanded. Cartridge slots are `<primary external="true" slot="N"/>`. Page mapping derived from `<mem base size>`. Classification by XML element type + `id` attribute.) (Date: 2026-03-29)
- [x] [Affects: problem-description-slotmap.md] What is the full set of device strings encountered across all target openMSX XML files? (Answer: ROM, RAM, MemoryMapper, PanasonicRAM, WD2793, TC8566AF, MSX-MUSIC, FMPAC, MSX-RS232, MSX-JE (as ROM id). MegaRam/SCCplus/SunriseIDE only in excluded Boosted configs. Starter LUT is sufficient.) (Date: 2026-03-29)
- [x] [Affects: problem-description-slotmap.md] Should numbered cartridge slots (CS1, CS2, …) be auto-generated from a single parameterised pattern or listed individually? (Answer: Parameterised — slot number read from `slot="N"` attribute on `<primary external="true">`. One rule, N derived from data.) (Date: 2026-03-29)
- [x] [Affects: problem-description-slotmap.md] Is the msx.org "Slot Map" HTML structure consistent across model pages? Manually check 5–10 pages. (Answer: Yes — surveyed all 137 non-category mirror pages (2026-04-10). Structure is consistent: `<h2><span id="Slot_Map" class="mw-headline">` introduces the section; the first `<table>` sibling contains the slot grid. Table has 5 or 6 rows (5-row = headers+4 page rows; 6-row = group header row + sub-slot header row + 4 page rows). Page rows top-to-bottom = C000h (P3) → 8000h (P2) → 4000h (P1) → 0000h (P0). Non-expanded slots are single data columns; expanded slots have one column per sub-slot. Cells use `rowspan` to merge across pages when a device spans multiple address ranges. 108/137 pages have a slot map; 29 do not (absent = no slot map data for that model). 4 pages have multiple slot maps (id suffix `_for_the_default_configuration`, `_for_the_upgraded_configuration`, `_2`) — parser should use the first (default). Parsing requires standard HTML table-flattening (tracking rowspan occupancy per column). Column header text is unreliable (mixed hyphens, varied wording); column position should be used instead.) (Date: 2026-04-10)
